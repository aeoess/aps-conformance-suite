# Contributing to aps-conformance-suite

This repo publishes the conformance fixtures for the Agent Passport System: deterministic vectors, schemas, and independent verifiers that let anyone check an implementation against the protocol from the bytes up, with no trust in us or in each other. That property only holds if the suite itself stays deterministic, recomputable, and precisely scoped, so contributions here run under a tighter policy than a typical fixture repo.

## The fastest way to contribute: run it and report

The most valuable contribution is a recompute or breakage report. Clone cold, run `validate.py` and `verify.ts`, and open an issue with your environment, the verbatim output, and what diverged. Reports of vectors that fail where they should pass, pass where they should fail, or diverge between the two verifiers get priority attention. No PR needed.

## What lands by PR

**Corrections to existing vectors and docs.** Wrong bytes, broken determinism, doc errors: open a PR with the failing recompute shown verbatim in the body. These merge quickly once we reproduce the failure.

**New vectors inside existing failure classes.** Open an issue first describing the case and which existing class and schema it exercises. A vector PR will be merged when:

1. **It is deterministic and recomputable.** Seeds recorded, canonical bytes produced under RFC 8785 JCS, and every check reproducible from the bytes up by both verifiers with no network access.
2. **It stays inside the existing schema and taxonomy.** No new fields, no new failure classes, no new verifier behavior. If your case needs any of those, it belongs in the section below, not in a vector PR.
3. **Generation is included.** The script or exact procedure that produced the vector ships with it, so the vector can be regenerated, not just replayed.
4. **Scope is one concern.** Fixture additions do not bundle schema edits, README changes, or verifier changes. Mixed PRs will be asked to split.

**External-system vector families** (fixtures generated by another implementation) land only after we have independently verified them end to end, and are labeled as ingestions from that system. Ask in an issue before building.

## What does not land by PR

Failure-class names, record fields, layer boundaries, verifier semantics, and normative README text are conformance vocabulary. Names published here are names downstream implementations bind to, so they are decided by the maintainer, as schema owner, on the maintainer's timeline. Proposals are welcome as issues and are genuinely read; a PR is not the vehicle for them, and a proposal that returns may return under different naming, placement, or none at all. PR quality does not change this: a technically perfect PR that mints vocabulary will be closed with thanks, not merged.

## What merging means

A merge means the vectors verified as deterministic, in scope, and correctly labeled. It is not an endorsement of the contributing system, and it implies nothing about that system's status, adoption, or production use. Please do not describe a merged fixture as APS validating, adopting, or partnering with anything.

## Mechanics

Fork, branch from `main`, PR against `main`. Both verifiers must pass locally before you open the PR; paste the verbatim output in the PR body. License is Apache 2.0 and contributions are accepted under the same terms.
